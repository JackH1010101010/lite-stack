## What You Already Have (and it's more than you think)

Your system is already 80% of a meta-orchestrator. Here's the honest audit:

What's working:
- Multi-agent consensus decision-making (Growth/Revenue/SEO perspectives)
- Registry-driven workflow dispatch (10 composable pipelines)
- Signal aggregation from 11 sources
- Experiment lifecycle (generate → measure → canary → promote)
- History-aware learning (history-summary.json regenerated by Claude)

What's missing — and this is the interesting part — isn't more complexity. It's three specific structural gaps.

---

## The Three Gaps

### Gap 1: Signal generation is decoupled from decision-making

Right now, signal generators (weekly-digest, hotel-refresh, explore, rate-monitor) run on their own schedule in deploy.yml. orchestrate.js reads whatever signals happen to exist when it runs on Wednesday.

This means the orchestrator makes decisions on stale data. If deploy.yml ran Sunday and orchestrate.yml runs Wednesday, signals are 3 days old. If the Sunday run failed, the orchestrator is blind.

**Fix: Unified signal phase.** orchestrate.js should run its own signals before deciding. Not all of them (some are expensive), but at minimum: rate-snapshot refresh, content-audit refresh, and a PostHog pull. The cheap ones that actually change the decision.

### Gap 2: No feedback loop from outcomes to strategy

The system learns what copy works (history.json tracks lift percentages). But it doesn't update its own strategy based on those learnings. STRATEGY-PACK.md is a static document. The multi-agent prompts use the same instructions every week regardless of what's been winning.

**Fix: Post-decision reflection.** After each experiment completes, a short Claude call summarises: "We tested X, got Y result, this means Z for future experiments." That gets appended to a machine-readable strategy block. The multi-agent prompts in Phase D pull from this evolving context.

### Gap 3: No distribution orchestration

The orchestrator manages the product (site content, copy, SEO, hotels) but not the distribution (Reddit posts, Google Ads, email). Distribution is manual. This is where 90% of the revenue impact is right now.

**Fix: Add distribution workflows to the registry.** A `reddit_distribute` workflow that checks which guides are published but not yet posted. A `google_ads_check` workflow that reads PostHog conversion data and suggests bid adjustments.

## The Mental Model

The right way to think about this isn't "build a meta-orchestrator." It's **OODA applied recursively**:

```
OBSERVE  →  ORIENT  →  DECIDE  →  ACT
   │           │          │         │
   │           │          │         └─ workflow-runner.js
   │           │          └─ multi-agent consensus
   │           └─ Phase C signal aggregation + strategy context
   └─ signal generators (weekly-digest, rate-monitor, etc.)
```

What makes it "meta" is that the OODA loop runs at multiple timescales:

- **Hourly**: Rate monitoring, site health checks (could be GitHub Actions cron)
- **Weekly**: The full orchestrate.js loop (measure → promote → decide → dispatch)
- **Monthly**: Strategic reassessment (explore new cities, audit content, refresh hotels)
- **Per-experiment**: The evolve.js lifecycle (generate → 7d measure → 7d canary → promote)

Each timescale feeds the one above it. Hourly checks produce signals. Weekly orchestration reads signals and acts. Monthly reassessment shifts strategy. Per-experiment results update the learning history.

You don't need a single "meta" script that controls everything. You need these four loops running independently but reading from the same shared state (experiments.json, history.json, rate-snapshot.json, strategy blocks).

## What to Build Next (Priority Order)

### 1. Distribution workflow in registry (highest ROI)

```json
{
  "reddit_distribute": {
    "name": "Reddit Guide Distribution",
    "description": "Post a published guide to relevant subreddits",
    "steps": [
      { "script": "autoresearch/distribute-reddit.js", "args": "--guide {{guide_id}}", "required": true }
    ],
    "signals_required": [],
    "cooldown_days": 3
  }
}
```

The script checks: which guides exist, which have been posted (track in distribution-log.json), picks the next unposted one, generates a context-appropriate post, and outputs it for manual review (or auto-posts via Reddit API if you set that up).

### 2. Inline signal refresh in orchestrate.js

Add to the top of Phase C:

```javascript
// Refresh cheap signals before reading
log('C', 'Refreshing rate snapshot...');
execSync('node autoresearch/rate-monitor.js --snapshot-only', { cwd: ROOT });
log('C', 'Refreshing content audit...');
execSync('node autoresearch/content-audit.js --quick', { cwd: ROOT });
```

This ensures the orchestrator never decides on stale local data. Keep the expensive signals (hotel-refresh, explore) on their monthly schedule.

### 3. Post-experiment reflection

Add to the end of Phase B (after a successful promotion):

```javascript
const reflection = await callClaude(
  `An A/B test just completed on ${exp.site}:
  Hypothesis: ${exp.hypothesis}
  Winner: ${winner.label} with ${lift}% lift
  What does this tell us about what works on this site?
  Return JSON: { "lesson": "...", "next_hypothesis": "...", "strategy_update": "..." }`
);
// Append to strategy-learnings.json
```

The multi-agent prompts in Phase D then include these learnings, so the system's taste evolves over time.

### 4. Adaptive scheduling

Right now everything is on fixed cron schedules. The system should adjust:
- If an experiment reaches significance early (day 4 instead of day 7), measure sooner
- If a distribution post gets traction, follow up faster
- If rate margins are dropping across the board, trigger rate-check more frequently

This is the most complex change and can wait. The simple version: add a `next_check` field to experiments.json that orchestrate.js reads, and have a daily lightweight cron that just checks "is anything due today?"

```
┌─────────────────────────────────────────────────────────────────┐
│                    META-ORCHESTRATOR v2                           │
│                                                                   │

│  ┌─── ACT ──────────────▼───────────────────────────────────┐    │
```

The key insight: the system is already recursive. orchestrate.js reads its own history to make better decisions. Adding the REFLECT phase just makes that loop tighter and more explicit.
